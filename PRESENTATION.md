# 基于大语言模型的交通时空知识图谱 — 答辩讲解文本

## 名词解释（放 PPT 最前面，先让听众听懂关键词）

| 术语 | 小白版（给同学讲） | 专业版（放 PPT 上） |
|------|-------------------|-------------------|
| **知识图谱** | 把信息组织成"实体-关系"网，像一张概念地图。比如"传感器A—位于—I-10高速"、"传感器A—附近有—加油站" | 以图结构组织知识的数据库，节点表示实体，边表示语义关系，支持高效的多跳关联查询 |
| **Neo4j** | 一种专门存图数据的数据库，不像MySQL存表格，它直接存点和连线 | 原生图数据库，采用属性图模型，Cypher查询语言天然适合表达图遍历和模式匹配 |
| **Cypher** | Neo4j的查询语言，写法很像自然语言，比如 `MATCH (s:Sensor) RETURN s` | Neo4j声明式图查询语言，基于ASCII art的模式匹配语法，对LLM生成友好 |
| **大语言模型(LLM)** | 就是DeepSeek、ChatGPT这类AI，你打字问它，它回你答案。我们的项目用它当"翻译官"——把中文翻译成数据库查询语句 | 基于Transformer架构的大规模预训练语言模型，本项目使用DeepSeek-chat完成Text-to-Cypher转换和结果自然语言生成 |
| **DCRNN** | 一种专门预测交通的AI模型。不同于只看历史的模型，它还看道路上下游的关系——前面堵了后面也会堵 | Diffusion Convolutional Recurrent Neural Network，将扩散卷积嵌入GRU门控单元，在有向道路图上同时建模时空依赖 |
| **Text-to-Cypher** | 你说中文问题 → AI自动生成数据库查询语句 → 查出结果 → 再用中文告诉你 | 自然语言到图查询语言的语义解析流水线，核心挑战是保证生成Cypher的语法正确性和语义一致性 |
| **Schema注入** | 每次问问题前，先把数据库的"使用说明书"发给AI，让它知道有哪些数据可以查 | 将目标数据库的结构元信息（节点标签、属性名、关系类型）作为上下文前置嵌入LLM的prompt |
| **METR-LA** | 洛杉矶高速公路207个测速器，2012年3-6月每5分钟记录一次，学术界最常用的交通数据集 | 洛杉矶县高速公路环形探测器网络数据集，34,272个时间步×207个传感器，是交通预测领域标准基准 |
| **POI** | Point of Interest —— 你地图上能搜到的餐厅、加油站、学校这些地点 | 地理兴趣点，通过osmnx从OpenStreetMap获取，用作传感器周边环境语义标签 |
| **osmnx** | 一个能自动下载开源地图数据的Python工具包，我们用它拿LA的路网和周边设施 | 基于Overpass API的OpenStreetMap数据下载与路网分析库 |
| **扩散卷积** | 可以理解为"让信息沿着道路图扩散出去"的数学方法——上游堵了，这个信息会沿着路网传到下游的传感器 | 将图上的多步随机游走作为卷积核，在有向图上做批量矩阵乘，同时捕获出度和入度两个方向的扩散过程 |
| **K-Means聚类** | 自动把207个传感器分成几组，让同一组里的传感器交通模式相似——比如"全天通畅型"和"高频拥堵型" | 基于欧氏距离的无监督聚类算法，将传感器按7维交通特征（速度、拥堵率、早晚高峰等）分为4类 |
| **UNWIND批量写入** | 本来一条一条往数据库里存，要发几百次请求。现在把所有数据打包，一次请求全写完 | 利用Cypher的UNWIND语法将列表反解为行，将N次单条写入合并为1次批量事务 |
| **bore隧道** | 一个工具，把你自己电脑上跑的服务暴露到公网，这样别人用手机也能访问 | 基于TCP端口转发的轻量级内网穿透工具，无需注册即可获得公网URL |

## 主线：LLM × 知识图谱 = 人人都能问交通问题

### 一句话讲清楚项目

"传统的交通系统只能给专家看数字。我们做的事情就是把交通传感器变成一个知识图谱，然后用大语言模型当翻译官——你只需要用中文问问题，系统就能自动帮你查数据库、给你答案，还能告诉你这个答案是怎么来的。"

### PPT素材（主标题）

> **核心命题**：如何让非技术人员也能从交通大数据中获得洞察？
>
> **方案**：将异构交通数据统一建模为**知识图谱**，以**大语言模型**作为自然语言接口，实现 Text-to-Cypher 的智能查询与分析。

### 主线流程图

```
非结构化数据（文本/新闻/报告）
        │
        ▼  LLM提取
┌───────────────────┐
│   Neo4j 知识图谱   │  ←── DCRNN预测结果（支线）
│   466节点 2151边  │  ←── 道路类型/POI（支线）
│   24种传感器属性   │  ←── 交通趋势/聚类（支线）
└────────┬──────────┘
         │
         ▼  LLM Text-to-Cypher
┌───────────────────┐
│  中文问答 + 可视化  │
│  "最堵的传感器在哪" │
│  → 答案 + 地图 + 来源│
└───────────────────┘
```

## 主线一：为什么要用知识图谱？

### 给同学讲

"一个传感器有几十个属性：经纬度、均速、拥堵率、道路类型、周围设施、历史趋势、预测误差……这些数据如果放在Excel里就是一张大表，问'和传感器A模式相似的传感器有哪些'需要用VLOOKUP写半天。但知识图谱天然擅长存储这种网状关系——传感器之间有道路连接、有模式相似、有周边设施，查起来又快又直观。

更重要的是，知识图谱的查询语言叫Cypher，写法很接近自然语言，比如 `MATCH (s:Sensor) WHERE s.road_type='motorway' RETURN s.sid`。大语言模型很容易学会写这种语句——比让它写SQL靠谱得多。"

### PPT素材

| 传统数据库 | Neo4j 图数据库 |
|-----------|---------------|
| 数据存在表格里，查关系要JOIN | 数据就是点和线，关系天然存在 |
| 多跳查询（A的朋友的朋友）写SQL很痛苦 | 一句MATCH走到底 |
| SQL语法复杂，LLM容易写出错的语句 | Cypher接近自然语言，LLM友好 |
| 传感器存成一行，属性平铺 | 传感器是节点，属性灵活扩展 |

**我们的KG里有什么：**

```
Sensor节点（207个，24个属性）
  ├ 位置: lat, lon
  ├ 道路: road_type (高速公路/匝道/主干道/三级路)
  ├ 预测: mae_15min, mae_30min, mae_60min
  ├ 趋势: avg_speed, congestion_ratio, peak_drop,
  │       morning_avg, evening_avg, weekend_avg...
  ├ 天气: rain_speed_drop, rain_congestion_increase
  ├ 聚类: cluster (全天通畅/高峰敏感/高频拥堵/中等波动)
  └ 描述: description (LLM生成的中文描述)

POI节点（259个）
  └ name, type (餐厅/加油站/学校/商场...)

关系（2,151条）
  ├ ROAD_DISTANCE {km} — 传感器间道路距离
  ├ NEAR {distance_m} — 传感器500m内的周边设施
  └ SIMILAR_TO {cluster} — 同一交通模式的相似传感器
```

## 主线二：LLM 怎么"理解"数据库？

### 给同学讲

"LLM本身不认识我们的数据库，不知道我们有哪些传感器、每个传感器有什么属性。所以每次提问时，我们会先把数据库的'说明书'提前发给LLM。

你可以理解为：你问LLM'帮我查最堵的路'，我们同时在后台偷偷塞给它一张小抄，上面写着'数据库里的传感器表有这些列：sid是编号、congestion_ratio是拥堵率、road_type是道路类型……下面给你几个例子参考'。LLM拿着这张小抄，就能写出正确的查询语句了。"

### PPT素材

**Text-to-Cypher 流水线：**

```
用户输入："最堵的10个传感器在哪？"
        │
        ▼
┌──────────────────────────┐
│ Step 1: 关键词匹配        │
│ "最堵" → 预写Cypher模板    │
│ 80%常见问题直接命中        │
└──────────┬───────────────┘
           │ (未匹配)
           ▼
┌──────────────────────────┐
│ Step 2: Prompt 组装        │
│ Schema说明书 + 5条示例 +   │
│ 属性速查表                 │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ Step 3: DeepSeek生成Cypher │
│ → 执行 → 结果格式化        │
│ → 中文回答 + 追问建议       │
└──────────────────────────┘
```

**为什么LLM能靠谱地生成Cypher？**

| 机制 | 小白解释 | 专业说明 |
|------|---------|---------|
| Schema注入 | 先发小抄给AI，让它知道有哪些数据能查 | 将节点标签、属性、关系类型作为前置context嵌入prompt |
| Few-shot示例 | 给5个例子当参考，AI有样学样 | 预置5条中英文对照的(问题→Cypher)示例对 |
| 关键词模板 | 拥堵/速度/天气等80%常见问题直接用预写好的查询 | 正则+关键词匹配覆盖高频查询维度 |
| 参数化查询 | 数据值用`$参数`代替，防止AI乱写数字 | 全部使用Neo4j参数化语法，杜绝注入和语法错误 |

## 支线一：DCRNN 预测 → 给 KG 提供预测数据

### 给同学讲

"LLM查的知识图谱里有一个很重要的维度——预测误差。比如你可以问'预测60分钟后最不准的传感器在哪'。这个'预测'就是从DCRNN深度学习模型来的。模型用了扩散卷积技术，能同时考虑时间规律（早晚高峰）和空间关系（上下游影响）。它预测未来15-60分钟的速度，我们把预测误差存到知识图谱里，LLM就能查了。"

### PPT素材

| 预测时长 | MAE (mph) | MAPE (%) | RMSE (mph) |
|----------|-----------|----------|------------|
| 15 min | 2.42 | 5.61 | 6.21 |
| 30 min | 2.97 | 6.78 | 7.90 |
| 60 min | 5.61 | 11.88 | 14.19 |

> **与主线的关系**：DCRNN的预测误差（mae_15min/mae_30min/mae_60min）作为Sensor属性存入KG，LLM的Schema包含它们，从而能回答"预测最不准的路段"、"哪种道路类型预测误差最大"等问题。

## 支线二：地理语义富化 → 让 KG 更丰富

### 给同学讲

"光有传感器坐标太单薄了。我们用OpenStreetMap的免费地图数据给每个传感器加了背景信息：它是在高速公路上还是匝道上？周围500米内有餐厅吗、有加油站吗？附近学校和商场多不多？

另外我们还分析了每个传感器的历史交通数据，算出了它的'性格'：是全天通畅型、还是早晚高峰堵、还是常年拥堵型。然后再把性格相似的传感器用线连起来。

这些信息全部存进知识图谱之后，LLM就能回答各种跨界问题了——'高速上的预测误差和普通道路比谁大'、'学校附近的传感器早上几点最堵'。"

### PPT素材

| 富化维度 | 写入的属性 | 实现了什么 |
|----------|-----------|-----------|
| osmnx道路匹配 | road_type | 按道路类型分类查询 |
| osmnx POI下载 | NEAR关系（259个POI） | 按周边设施筛选传感器 |
| 历史趋势统计 | avg_speed, congestion_ratio, peak_drop等10项 | 拥堵/速度/高峰模式分析 |
| K-Means聚类 | cluster + SIMILAR_TO关系 | 发现相似交通模式的传感器 |
| LLM描述 | description | 30字中文概括每个传感器 |

> **与主线的关系**：这些属性全是KG中Sensor节点的字段，LLM做Text-to-Cypher时Schema里会包含它们，从而让LLM能回答的问题类型从几十种扩展到几百种。

## 支线三：工程优化 → 让演示跑得快

### 给同学讲

"还有两个工程上的改进很重要：一是我们用bore工具把本地服务暴露到公网，这样带个笔记本开热点，评委拿手机就能直接访问我们的系统。二是所有数据写入都做了批量优化——本来往数据库里一条一条存，要发几百次请求，现在一次打包全写完，速度快了几百倍。"

### PPT素材

| 优化项 | 优化前 | 优化后 | 提升 |
|--------|--------|--------|------|
| 创建传感器节点 | 207次写入 | 1次UNWIND批量 | ~200× |
| 创建道路距离边 | 1035次写入 | 1次UNWIND批量 | ~1000× |
| 趋势指标写入 | 207次写入 | 1次UNWIND批量 | ~200× |
| 趋势计算 | 逐传感器循环×207 | numpy矩阵向量化 | ~100× |
| POI匹配写入 | ~700次写入 | 3次UNWIND批量 | ~230× |
